
# REALITY CHECK 


● ¿Qué tareas subestimamos y por qué? (Falta de conocimiento, problemas de 
entorno, etc.). 

Desde el principio se tenía claro que este sería un MVP pequeño y muy enfocado, por lo que las historias de usuario estaban bien definidas. Se tomaron las primeras cuatro porque eran las más importantes para construir el motor de detección de fraude. Por esta razón, no se considera que haya habido una subestimación de tareas, ya que el trabajo a realizar era claro y manejable.

Sin embargo, se identificó que dentro de esas tareas se incluyeron algunas actividades que no eran necesarias para esta primera versión, especialmente aquellas relacionadas con la base de datos. Para este MVP, lo más importante era construir el motor de reglas y validar la lógica, sin necesidad de implementar persistencia de información.

En ese sentido, más que una subestimación, lo que ocurrió fue la inclusión de tareas que no eran necesarias en ese momento. Esto permite como aprendizaje que, en futuras iteraciones, se debe priorizar únicamente lo esencial para cada MVP, enfocándose en la lógica principal y dejando otros componentes para etapas posteriores donde realmente aporten valor.
